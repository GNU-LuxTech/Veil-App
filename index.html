// core_principles

Built on non-negotiable privacy

Every architectural decision in Veil flows from five immutable principles.

🔒
Zero Persistence

Messages exist only in RAM. No logs, no database, no disk writes. The moment a session ends, the exchange is gone.

👤
Zero Identity

No phone number, no email, no username registry. Your Ed25519 keypair is your identity — nothing more.

🌐
Zero Servers

No central infrastructure at all. Tor handles all routing. There is nothing to seize, no data to subpoena.

🛡️
Zero Trust

Every incoming connection requires explicit human acceptance. Nothing is automatic, nothing is assumed.

⚡
Synchronous Only

Both parties must be online simultaneously. No offline delivery by design — this is a feature, not a limitation.

// how_it_works

Simple flow, zero exposure

Veil handles all the cryptographic complexity. You just connect and talk.

01
Generate your identity

On first launch, Veil generates an Ed25519 keypair. Your Tor v3 onion address is derived directly from your public key — permanent, self-authenticating, unfakeable.

02
Share your address out-of-band

There is no discovery server. Share your 56-character onion address over Signal, on paper, or spoken aloud. Veil does not broadcast your presence.

03
Accept or reject incoming connections

When a peer connects, Veil prompts you. You decide. Every session is opt-in — no silent connections, no background listeners.

04
Encrypted session, then silence

X25519 ECDH key exchange derives a session key. ChaCha20-Poly1305 encrypts the stream. When you disconnect, the key is gone forever.

// encryption_model

Layered security,
no complexity for you

Veil stacks multiple layers of cryptographic protection. Tor provides transport-level anonymity; your session key provides end-to-end confidentiality.

✓ No Double Ratchet needed — sessions are synchronous
✓ Session key exists only in RAM, never on disk
✓ Onion address authenticates peer — no MITM possible
✓ Neither party sees the other's real IP address
🌐
Tor Network — Transport Layer
3-hop onion routing. Neither peer sees the other's IP.
↓
🔑
X25519 ECDH — Key Exchange
Derives a symmetric session key on connection. Forward-secret by design.
↓
🔐
ChaCha20-Poly1305 — AEAD Cipher
Encrypts every message with authentication. Modern, fast, audited.
↓
💨
Session End — Cryptographic Amnesia
Session key wiped from memory. No record kept, anywhere.

// tech_stack

Engineered for the task

Every dependency was chosen deliberately. Nothing bloated, nothing unnecessary.

Layer Technology Rationale
Language Go Native Tor support via bine, strong concurrency primitives, single binary output
Tor Integration bine Embeds Tor directly — no external Tor installation required from the user
Onion Services Tor v3 (Ed25519) Self-authenticating, NAT-transparent addresses. Permanent identity, zero registration
Key Exchange X25519 ECDH Fast, secure, industry-standard. Provides forward secrecy per session
Symmetric Cipher ChaCha20-Poly1305 Modern AEAD cipher, constant-time, mobile-friendly, hardware-independent
Terminal UI bubbletea + lipgloss Termux-style TUI framework. The terminal IS the product
Key Storage Local file (0600) Private key never leaves the device. No cloud sync, no backup service

Where we are, where we're going

Two phases complete. Two phases ahead — features and hardening.

Phase 01 Complete
Core
  • Keypair generation on first launch
  • Embedded Tor daemon via bine
  • Onion service creation & display
  • Outbound connection to peer address
  • Inbound listener with accept/reject
  • X25519 session key exchange

Phase 02 Complete
Master UI
  • Interactive TUI Main Menu (Host vs Join)
  • Graceful CLI flag bypass (hybrid mode)
  • Loading screens (Tor startup, onion creation)
  • Address input form
Phase 03 In Progress
Features
  • /send file transfer over active session
  • Contact alias system (local nicknames)
  • --auto-accept whitelist for trusted addresses
  • Session transcript opt-in (encrypted, local only)
  • Multi-platform builds (Linux, macOS, Windows, Termux)
Phase 04 Planned
Hardening
  • Traffic padding (resist timing analysis)
  • Canary mode (detect binary tampering)
  • External security audit
  • Source-available public release
